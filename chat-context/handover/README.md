# handover

The return folder for the Code apparatus. **Every** Code dispatch ends by
writing its report here — there are no unregulated Code dispatches. The Desktop
(concept) session curates these reports into the durable record at the next
closure.

## The rule (WORKING_CONCEPT §14)
- A Code sub-sprint's CLOSURE **must** write a structured report to
  `chat-context/handover/handover-<YYYY-MM-DD>-<slug>.md` — the result, what was
  built/measured, what passed the gate, what was deferred, and any new finding or
  decision that needs Concept ratification.
- Code still returns a **one-line pointer** to the Desktop session ("done, report
  here: `<path>`") as the signal — but the substance lives in the handover file,
  never only in chat.
- Desktop, at the next closure, **curates** each handover report into the durable
  record (`sprints/` sprint file, `findings/`, or a decision note), then prunes
  the consumed report. This folder is a **staging area**, not the durable record.

## Discipline
- Nothing here is authoritative on its own — the curated record is.
- A report that has been folded into a sprint file / finding is removed (consumed).
- Naming: `handover-<YYYY-MM-DD>-<slug>.md`, chronological and self-describing.

> The Code apparatus cannot write mnemonics or ledgers; anything newly decided in
> a Code sub-sprint returns here and is ratified by Concept (WORKING_CONCEPT §8, §14).
